# 법적 검토 결과 및 구조 변경 사유

> 작성일: 2026-01-29

## 1. 왜 구조를 바꿔야 하나?

### 원래 계획 (포인트 시스템)
```
후원자 → 포인트 충전(1:1) → 플랫폼 보관 → 개발자에게 주급 → 포인트 환전(1:1)
```

### 문제: 이 구조는 **불법**

| 행위 | 위반 법령 | 처벌 |
|-----|---------|-----|
| 1:1 환전 가능 포인트 발행 | 전자금융거래법 (미등록 선불업) | 징역 3년 |
| 투자 + 주급(수익) + 원금보장 | 유사수신행위법 | 징역 5년 |
| 플랫폼이 정산 자금 보관 | 전자금융거래법 (미등록 PG) | 징역 3년 |

**핵심**: 플랫폼이 돈을 만지면 = 금융업 = 등록 필요 = 자본금 수십억 필요

---

## 2. 해결책: PG 에스크로 활용

### 변경된 구조
```
후원자 → PG사 에스크로 계좌 → (조건 충족 시) → 개발자 계좌
              ↑
         플랫폼은 "지급 승인" 신호만 보냄
         돈은 절대 안 만짐
```

### 왜 합법인가?
- **플랫폼이 돈을 보관하지 않음** → 선불업/PG업 등록 불필요
- **투자가 아닌 후원** → 증권법/유사수신법 적용 안 됨
- **이미 검증된 방식** → 텀블벅, 와디즈, 크몽 등이 동일 구조 사용

---

## 3. 용어 변경

| 기존 | 변경 | 이유 |
|-----|-----|-----|
| 투자자 | **후원자/서포터** | "투자"는 자본시장법상 증권 발행에 해당 |
| 투자 | **후원** | 수익 기대 X, 리워드만 제공 |
| 주급 | **정산금/기성금** | "급여"는 고용관계 의심받음 |
| 포인트 충전 | **결제** | 직접 PG 결제, 포인트 없음 |
| 포인트 환전 | **환불** | 조건부 환불만 가능 |

---

## 4. 자금 흐름 비교

### 기존 (불법)
```
[후원자] --현금--> [플랫폼 계좌] --주급--> [개발자]
                      ↑
                 플랫폼이 보관
                 = 금융업 = 불법
```

### 변경 (합법)
```
[후원자] --결제--> [PG 에스크로] --정산--> [개발자]
                       ↑
                  PG사가 보관
                  플랫폼은 API 호출만
```

---

## 5. 실제 사례

| 플랫폼 | 구조 | 비고 |
|-------|-----|-----|
| 텀블벅 | 후원금 → PG 에스크로 → 창작자 정산 | 리워드형 |
| 와디즈 | 펀딩금 → PG 에스크로 → 메이커 정산 | 리워드형 |
| 크몽 | 결제금 → PG 에스크로 → 전문가 정산 | 용역 중개 |
| 숨고 | 결제금 → PG 에스크로 → 전문가 정산 | 용역 중개 |

**모두 "플랫폼이 돈 안 만지는" 구조**

## 6. 사용할 PG 서비스

| 서비스 | 기능 | 비고 |
|-------|-----|-----|
| 토스페이먼츠 | 에스크로 + 정산 API | 국내 1위 |
| 포트원 | 여러 PG 통합 + 정산 대행 | 개발 편의성 |
| NHN KCP | 에스크로 서비스 | 안정성 |

---

## 7. 플랫폼 수익 모델 변경

### 기존
- 개발자 환전 시 10% 수수료

### 변경
- **후원 성사 시 중개 수수료 (7~10%)**
- PG사가 정산할 때 수수료 제외하고 지급

```
후원금 100만원
├─ 개발자 정산: 90만원
├─ PG 수수료: 3만원
└─ 플랫폼 수수료: 7만원
```

---

## 8. 결론

| 항목 | 기존 | 변경 |
|-----|-----|-----|
| 돈 보관 주체 | 플랫폼 | PG사 |
| 법적 리스크 | 형사처벌 가능 | 없음 |
| 등록 필요 | 선불업 + PG업 | 통신판매중개업 (신고제) |
| 자본금 요건 | 수십억 | 없음 |

**이 구조로 가야 법적으로 안전하게 서비스 운영 가능**
